Cost Structure Anatomy

任何优化策略的前提是理解成本的构成。

Claude Code 的费用来自 Anthropic API 调用，按 token 计价。截至 2025 年的公开定价：

模型输入（每百万 token）输出（每百万 token）
Claude Opus 4$15$75
Claude Sonnet 4$3$15
Claude Haiku 3.5$0.80$4
值得注意的不对称性：输出单价是输入的 5 倍，而 Opus 的输出单价达到了 Sonnet 的 5 倍。这意味着一次让 Opus 生成长篇分析的操作，成本可能相当于 25 次 Sonnet 的等长输入。

单次交互（分析一个文件并修改它）的 token 消耗通常在 10K-50K 之间，费用从几美分到几十美分不等。这个数字本身不吓人。真正的成本陷阱在别处。

上下文膨胀是隐形的费用黑洞。Claude Code 的每一轮对话都将完整的对话历史发送给 API。到第 20 轮时，你发送的一条简短指令实际上附带了前面 19 轮的全部内容。对话越长，每一轮的边际成本越高——这是一条向上弯曲的成本曲线。

养成使用 /cost 命令的习惯。它提供当前会话的累计 token 消耗和费用。持续的费用感知是一切优化行为的起点。

The prerequisite for any optimization strategy is understanding cost composition.

Claude Code costs come from Anthropic API calls, billed per token. Public pricing as of 2025:

ModelInput (per million tokens)Output (per million tokens)
Claude Opus 4$15$75
Claude Sonnet 4$3$15
Claude Haiku 3.5$0.80$4
Note the asymmetry: output unit price is 5x input, and Opus output is 5x Sonnet output. This means a single Opus long-analysis generation can cost the equivalent of 25 Sonnet equal-length inputs.

A typical interaction (analyzing and modifying a file) consumes 10K-50K tokens, costing a few cents to a few dozen cents. This figure alone is not alarming. The real cost trap lies elsewhere.

Context inflation is the invisible cost black hole. Each conversation round in Claude Code sends the complete conversation history to the API. By round 20, your brief instruction actually carries all 19 previous rounds of content. The longer the conversation, the higher the marginal cost per round — an upward-curving cost function.

Build the habit of using /cost. It provides cumulative token consumption and cost for the current session. Continuous cost awareness is the starting point for all optimization behavior.

Strategy 1: Context Compression

/compact 是 Claude Code 最重要的成本控制工具。它将当前对话历史压缩为一段简短摘要，用摘要替换完整历史。

触发时机的经验判断：

  • 对话超过 10 轮
  • 刚完成一个子任务，即将进入下一个
  • /cost 显示费用增速明显抬头
一次 /compact 通常能将上下文从数万 token 压缩至数千 token。此后每一轮对话都因此受益——这是一项具有持续复利效应的操作。

Claude Code 也支持自动压缩。可以在设置中配置当上下文达到一定阈值时自动触发 compact（具体配置方式请参考官方文档，因版本更新可能有所调整）。

/compact is Claude Code's most important cost control tool. It compresses current conversation history into a brief summary, replacing the full history with that summary.

Heuristics for when to trigger:

  • After 10+ conversation rounds
  • Just completed a subtask, about to start the next
  • /cost shows noticeably accelerating costs
A single /compact typically compresses context from tens of thousands of tokens to a few thousand. Every subsequent round benefits — an operation with sustained compound returns.

Claude Code also supports automatic compression. You can configure it to auto-trigger compact when context reaches a certain threshold (refer to official documentation for specific configuration, as it may change across versions).

Strategy 2: Model-Tiered Scheduling

并非所有任务都需要动用最强模型。这个认知偏差是大多数用户超支的首要原因。

通过 /model 命令可以随时切换模型。基于任务复杂度的分级调度策略：

Sonnet 即可胜任的任务：
  • 单文件 bug 修复
  • 代码解释与文档生成
  • 测试代码编写
  • 格式化、重命名等机械操作
  • 翻译任务
Opus 不可替代的任务：
  • 跨多文件的复杂重构
  • 需要全局架构理解的功能开发
  • 深层调用链的调试
  • 需要创造性方案的设计问题
Sonnet 的单价是 Opus 的五分之一。如果 60% 的任务可以分流给 Sonnet，仅此一项策略即可节省近 50% 的总开销。这不是估算——这是简单的算术。

Not every task requires the most powerful model. This cognitive bias is the primary cause of most users' overspending.

Use /model to switch models at any time. A tiered scheduling strategy based on task complexity:

Tasks Sonnet handles well:
  • Single-file bug fixes
  • Code explanations and documentation
  • Test code generation
  • Mechanical operations (formatting, renaming)
  • Translation tasks
Tasks requiring Opus:
  • Complex refactoring across multiple files
  • Feature development requiring global architectural understanding
  • Debugging deep call chains
  • Design problems requiring creative solutions
Sonnet's unit price is one-fifth of Opus. If 60% of tasks can be routed to Sonnet, this strategy alone saves nearly 50% of total spending. This is not an estimate — it is simple arithmetic.

Strategy 3: Conversation Granularity Control

一个大任务不应该在一个连续对话中从头执行到尾。原因已在"上下文膨胀"一节中阐明：对话越长，每一轮的边际成本越高。

更优的策略是将任务分解为独立子任务，每个子任务使用一个新对话（或 /clear 重置后的干净上下文）。

以"为项目增加用户管理模块"为例，可分解为：

    • 对话 1：设计数据库 schema 和 API 接口（Opus，需要架构判断）
    • 对话 2：实现数据库 migration 和 service 层（Sonnet，执行性任务）
    • 对话 3：实现 API 路由（Sonnet）
    • 对话 4：编写测试（Sonnet）
    • 对话 5：集成测试和 bug 修复（Opus，需要跨模块理解）
每个对话从干净上下文启动，规避了膨胀成本。前序对话的设计产出可以直接粘贴到后续对话作为输入——信息传递的成本远低于历史膨胀的成本。

A large task should not execute from start to finish in a single continuous conversation. The reason was articulated in the "context inflation" section: the longer the conversation, the higher the marginal cost per round.

The superior strategy is decomposing tasks into independent subtasks, each using a new conversation (or a clean context after /clear).

Taking "add a user management module" as an example:

    • Conversation 1: Design database schema and API interfaces (Opus — requires architectural judgment)
    • Conversation 2: Implement database migration and service layer (Sonnet — execution task)
    • Conversation 3: Implement API routes (Sonnet)
    • Conversation 4: Write tests (Sonnet)
    • Conversation 5: Integration testing and bug fixes (Opus — requires cross-module understanding)
Each conversation starts from clean context, avoiding inflation costs. Design outputs from earlier conversations can be pasted directly into later ones — information transfer costs are far lower than history inflation costs.

Strategy 4: Input Precision

模糊指令是 token 的黑洞。一条"帮我优化一下这个项目的性能"会触发 Claude 扫描整个代码库——读取大量不相关的文件、生成冗长的分析报告，消耗数万 token 却可能命中率极低。

Vague instructions are token black holes. A "help me optimize this project's performance" triggers Claude to scan the entire codebase — reading volumes of irrelevant files, generating verbose analysis reports, consuming tens of thousands of tokens with potentially minimal hit rate.

Compare these two inputs:

前者的 token 消耗可能是后者的 10 倍以上，而产出的可用性可能更低。

一条实用原则：当你已知目标文件的路径时，在指令中直接声明它。 这一条简单的习惯能消除 Claude 搜索和定位环节的全部开销。

The former's token consumption can be 10x or more than the latter, while its output usability may be lower.

A practical principle: when you know the target file path, declare it directly in your instruction. This simple habit eliminates all overhead from Claude's search and navigation phase.

Strategy 5: Subagent Isolation

Claude Code 的 Agent 工具可以派出子代理执行独立任务。子代理拥有自己独立的上下文窗口，其 token 消耗不会回流至主对话的上下文。

这提供了一种精妙的成本控制手段：将需要大量读取操作的调研性工作（如"检查这个代码库的测试框架配置"）委派给子代理，主对话仅接收最终的简洁摘要。调研过程中消耗的数万 token 被隔离在子代理的上下文中，不会导致主对话的上下文膨胀。

Claude Code's Agent tool can dispatch subagents for independent tasks. Subagents have their own context windows, and their token consumption does not flow back into the main conversation's context.

This provides an elegant cost control mechanism: delegate read-heavy investigative work (e.g., "examine this codebase's test framework configuration") to subagents, while the main conversation receives only the final concise summary. The tens of thousands of tokens consumed during investigation are isolated in the subagent's context, preventing main conversation inflation.

Core Logic of Cost Control

归结起来就是两条原理：

    • 减少不必要的 token 消耗——compact、精准输入、子代理隔离
    • 用更低单价的模型完成可胜任的任务——模型分级调度
五个策略按投入产出比排序：
    • 模型分级调度（即时生效，效果最显著）
    • 上下文压缩（养成习惯即可）
    • 输入精度（需要刻意训练）
    • 对话粒度控制（大型任务必备）
    • 子代理隔离（进阶技巧）
不必一次全部部署。从模型切换和 compact 起步，这两项即可覆盖绝大部分节省空间。

It reduces to two principles:

    • Eliminate unnecessary token consumption — compact, precise input, subagent isolation
    • Route tasks to lower-priced models that can handle them — model-tiered scheduling
Five strategies ranked by ROI:
    • Model-tiered scheduling (immediate effect, most significant)
    • Context compression (just build the habit)
    • Input precision (requires deliberate practice)
    • Conversation granularity control (essential for large tasks)
    • Subagent isolation (advanced technique)
No need to deploy all at once. Starting with model switching and compact covers the majority of savings potential.
