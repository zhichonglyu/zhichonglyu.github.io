Claude Code 的基础功能——读写文件、运行命令、对话式交互——已经能覆盖大部分日常编程任务。但真正将其从"一个好用的工具"转化为"一套完整工作流"的，是下面这些进阶机制。它们的共同特征是：将人类的重复性判断固化为自动化规则。

Claude Code's basic capabilities — reading/writing files, running commands, conversational interaction — cover most daily programming tasks. But what truly transforms it from "a useful tool" into "a complete workflow" are the following advanced mechanisms. Their shared characteristic: crystallizing repetitive human judgments into automated rules.

Hooks: Programmatic Behavior Constraints

Hooks 允许你在 Claude 执行特定工具调用的前后，自动运行 shell 命令。这不是一个锦上添花的功能——它从根本上改变了你与 AI 代理之间的信任模型。

三种典型应用模式：

自动格式化。Claude 生成的代码未必完全符合你的 Prettier/ESLint 配置。PostToolUse hook 在每次文件修改后自动格式化，将代码风格的一致性从"Claude 的努力"升级为"系统保证"。 自动测试。每次编辑测试文件后自动运行对应的测试套件，实现即时反馈循环。 硬性安全边界。这是 Hooks 最具架构意义的应用：

Hook commands can access context information through environment variables (refer to official documentation for specific available variables, as they may change across versions).

Three typical application patterns:

Auto-formatting. Code generated by Claude may not perfectly match your Prettier/ESLint configuration. A PostToolUse hook that auto-formats after each file modification upgrades code style consistency from "Claude's best effort" to "system guarantee." Auto-testing. Automatically run the corresponding test suite after each test file edit, creating an instant feedback loop. Hard safety boundaries. This is the most architecturally significant application of Hooks:

当 PreToolUse hook 返回非零退出码时，Claude 会中止该工具调用。这是一道不依赖于 AI 自律性的硬性防线——无论 Claude 的意图如何，敏感文件都不会被修改。

这一机制的意义超越了单纯的安全防护。它建立了一种人机协作的新范式：人类通过声明式规则定义边界，AI 在边界内自由行动。信任不再是一个二元选择（完全信任 vs 逐条审批），而成为一个可编程的连续体。

When a PreToolUse hook returns a non-zero exit code, Claude aborts that tool call. This is a hard boundary that does not rely on AI self-discipline — regardless of Claude's intent, sensitive files will not be modified.

The significance of this mechanism extends beyond mere security. It establishes a new paradigm for human-AI collaboration: humans define boundaries through declarative rules, and AI operates freely within those boundaries. Trust is no longer a binary choice (full trust vs. per-action approval) but becomes a programmable continuum.

MCP: Connecting Claude to External Systems

MCP（Model Context Protocol）是 Anthropic 推出的开放协议，使 AI 模型能够调用外部工具和服务。Claude Code 原生支持 MCP 服务器，意味着你可以将数据库、GitHub、Slack 等外部系统纳入 Claude 的操作范围。

社区已构建了覆盖面广泛的 MCP 服务器生态：

  • 数据库：PostgreSQL、MySQL、SQLite——直接查询和修改数据
  • GitHub：创建 issue、提交 PR、查看 CI 状态
  • Slack：读写消息
  • 浏览器自动化：通过 Puppeteer/Playwright 控制浏览器
  • 文件系统：更细粒度的文件访问控制
MCP 的战略价值在于它改变了 Claude Code 的能力边界。没有 MCP 的 Claude Code 是一个代码编辑代理；有了 MCP 的 Claude Code 是一个全栈工作台。你可以在一个对话中完成"查询数据库 → 分析结果 → 编写代码 → 提交 PR"的完整链路，无需在工具之间切换。

The community has built a broad MCP server ecosystem:

  • Databases: PostgreSQL, MySQL, SQLite — direct data queries and modifications
  • GitHub: Create issues, submit PRs, check CI status
  • Slack: Read and write messages
  • Browser automation: Control browsers via Puppeteer/Playwright
  • Filesystem: Fine-grained file access control
MCP's strategic value lies in redefining Claude Code's capability boundary. Without MCP, Claude Code is a code editing agent; with MCP, it becomes a full-stack workstation. You can complete the full chain of "query database → analyze results → write code → submit PR" in a single conversation, without switching between tools.

Memory: Cross-Session Knowledge Persistence

Claude Code 的 Memory 系统通过在 ~/.claude/ 目录下维护记忆文件，实现跨对话的偏好和知识持久化。

触发方式直截了当：

The trigger mechanism is straightforward:

Claude 将这些偏好写入记忆文件，后续新对话自动继承。

Memory 与 CLAUDE.md 的职责分野值得明确：

维度CLAUDE.mdMemory
作用域项目级跨项目
版本控制是（提交到 git）否（本地存储）
共享性团队共享个人私有
内容类型项目规范、技术栈、命令个人偏好、工作流习惯
二者是互补关系而非替代关系。CLAUDE.md 回答"这个项目如何运作"，Memory 回答"这个用户如何偏好"。

Memory 适合存储：个人编码偏好、工作流习惯、反复出现的问题和解决方案。不适合存储：临时任务状态、可能过期的信息、已经在 CLAUDE.md 中声明过的内容。

Claude writes these preferences to memory files, which are automatically inherited by subsequent conversations.

The division of responsibility between Memory and CLAUDE.md is worth clarifying:

DimensionCLAUDE.mdMemory
ScopeProject-levelCross-project
Version controlYes (committed to git)No (local storage)
SharingTeam-sharedPersonal/private
Content typeProject specs, tech stack, commandsPersonal preferences, workflow habits
The two are complementary, not substitutive. CLAUDE.md answers "how this project works"; Memory answers "how this user prefers to work."

Memory is suitable for: personal coding preferences, workflow habits, recurring problems and solutions. Not suitable for: temporary task states, potentially expirable information, content already declared in CLAUDE.md.

Git Workflow Integration

Claude Code 对 git 的支持不止于"能执行 git 命令"。它理解版本控制的语义结构，能在更高的抽象层次上协助 git 操作。

智能提交

Claude Code's git support goes beyond "can execute git commands." It understands the semantic structure of version control and assists at a higher level of abstraction.

Smart commits

Claude 执行 git diff，分析所有变更的语义（而非仅仅列出文件名），生成一条准确反映变更意图的 commit message，然后执行提交。在赶工期的日子里，这比你随手写的"fix bug"或"update code"有价值得多。

PR 创建

Claude runs git diff, analyzes the semantics of all changes (not merely listing filenames), generates a commit message that accurately reflects the intent of the changes, and executes the commit. During crunch time, this is far more valuable than a hastily written "fix bug" or "update code."

PR creation

Claude 分析分支上的全部 commit，生成包含变更摘要、影响范围和测试建议的 PR 描述，然后通过 gh CLI 完成创建。

代码审查

Claude analyzes all commits on the branch, generates a PR description including change summary, impact scope, and testing suggestions, then completes creation via gh CLI.

Multi-Agent Parallelism

处理大型任务时，Claude Code 可以同时派出多个子代理并行工作。要在 10 个文件中执行类似修改？Claude 可以启动多个子代理分别处理不同文件，而非串行逐一操作。

For large tasks, Claude Code can dispatch multiple subagents to work in parallel. Need similar modifications across 10 files? Claude can launch multiple subagents to handle different files simultaneously rather than processing them sequentially.

并行处理不仅更快，而且更经济——每个子代理的上下文独立运作，不会互相叠加膨胀。

Parallel processing is not only faster but more economical — each subagent's context operates independently, with no mutual inflation.

Assembly: A Complete Workflow

将以上机制组装起来，一个成熟的 Claude Code 工作流长这样：

    • CLAUDE.md 声明项目规范和命令（见本系列第二篇）
    • Hooks 保障代码风格一致性和安全边界
    • MCP 打通 GitHub 等外部系统
    • Memory 持久化个人偏好
配置到位后，日常流程简化为：打开终端 → claude → 自然语言描述任务 → 审查结果 → 下一个任务。

这套工作流对终端重度用户尤为自然。如果你的日常已经在 tmux/terminal 中度过，Claude Code 不要求你改变任何习惯——它只是让你的终端多了一个能理解自然语言的强力代理。

Combining the above mechanisms, a mature Claude Code workflow looks like:

    • CLAUDE.md declares project conventions and commands (see the second post in this series)
    • Hooks ensure code style consistency and safety boundaries
    • MCP connects to external systems like GitHub
    • Memory persists personal preferences
With configuration in place, the daily flow simplifies to: open terminal → claude → describe task in natural language → review results → next task.

This workflow is particularly natural for heavy terminal users. If your daily life already takes place in tmux/terminal, Claude Code requires no habit changes — it simply adds a powerful natural-language-understanding agent to your terminal.
